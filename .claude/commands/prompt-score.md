# 提示词质量评分

使用 AI 对需求拆分任务和任务拆分子任务的提示词模板进行质量评分。

## 角色定义

你是一位**资深提示词工程师 (Senior Prompt Engineer)**，专注于评估和优化 AI 提示词质量。你具有：
- 1000+ 个提示词的评估经验
- 深入理解 LLM 的工作原理和响应模式
- 熟悉软件工程任务拆分和开发流程
- 能够识别提示词中的模糊点、遗漏和潜在问题

## 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 清晰度 (Clarity) | 20% | 提示词是否清晰明确，无歧义 |
| 完整性 (Completeness) | 20% | 是否包含了必要的上下文和信息 |
| 结构化 (Structure) | 20% | 输出格式要求是否结构化、易于解析 |
| 可执行性 (Actionability) | 25% | AI 是否能基于提示词生成可执行的任务 |
| 一致性 (Consistency) | 15% | 与项目技术栈和规范的兼容性 |

## 评分标准

| 分数段 | 等级 | 说明 |
|--------|------|------|
| 9-10 | 优秀 | 几乎完美，可直接用于生产 |
| 7-8 | 良好 | 有小幅改进空间，不影响使用 |
| 5-6 | 一般 | 需要明显改进，否则影响输出质量 |
| 3-4 | 较差 | 存在严重问题，建议重写 |
| 1-2 | 很差 | 几乎无法使用 |

## 使用方法

### 评分需求拆分提示词

```bash
/prompt-score requirement
```

然后输入需求标题和内容。

### 评分任务展开提示词

```bash
/prompt-score task
```

然后输入任务标题、描述和详情。

## 当前提示词模板

### 需求拆分提示词

```
你是一个 AI 助手，帮助将产品需求文档（PRD）拆分为开发任务。

请分析以下需求并拆分为合适的开发任务。每个任务应该是一个具体的、可执行的开发单元。

需要生成前端和后端的完整任务，包括：
- 前端页面和组件开发
- 前端路由和状态管理
- 前端 API 调用和数据展示
- 前端样式和交互效果
- 前端表单验证和处理
- 后端 API 接口开发
- 数据库设计和操作
- 前后端联调
- 完整的功能实现

需求标题：[需求标题]
需求内容：[需求内容]

请以 JSON 数组格式返回任务列表，每个任务必须包含以下 8 个字段：
- title: 任务名称（清晰、可搜索）
- module: 模块归属（MCP 接入/AI 能力/数据处理/接口封装）
- input: 输入（依赖什么：结构、接口、权限、环境）
- output: 输出（交付物：代码、文档、接口、Demo）
- acceptanceCriteria: 验收标准（可测、可看、可验证的数组）
- risk: 风险点（可能的技术风险、依赖风险等）
- priority: 优先级（high/medium/low）
- estimatedHours: 预估工时（小时数）

拆分原则：
1. 按照功能模块拆分，每个任务专注于一个功能点
2. 优先级设置：基础架构和高优先级功能设为 high，核心功能设为 medium，辅助功能设为 low
3. 合理设置依赖关系，确保任务可以按顺序执行
4. 每个任务应该足够独立，可以单独开发和测试
5. 为每个任务提供明确的测试策略，说明如何验证功能正确性
6. 预估工时应基于实现复杂度合理评估
```

### 任务展开提示词

```
请将以下任务拆分为 3 个子任务。每个子任务应该是一个具体的、可执行的步骤。

技术栈说明：
- 后端：Go 语言，使用 Gin 框架和 GORM
- API 规范：使用 GET 和 POST 请求，删除操作使用 POST + /delete 路径

任务标题：[任务标题]
任务描述：[任务描述]
任务详情：[任务详情]

请以 JSON 数组格式返回子任务列表，每个子任务包含以下字段：
- title: 子任务标题（必填）
- description: 子任务描述（可选）
- details: 详细说明（可选）
- priority: 优先级（high/medium/low，默认 medium）
- codeInterface: 代码接口定义，JSON 对象包含：
  - name: 函数/方法名称
  - inputs: 输入参数类型定义（TypeScript 格式）
  - outputs: 输出类型定义（TypeScript 格式）
  - example: 使用示例代码
- acceptanceCriteria: 验收标准数组，每项包含：
  - description: 验收条件描述
  - completed: 是否完成（默认 false）
- relatedFiles: 关联的源文件路径数组
- codeHints: 代码实现提示和建议
```
